# 牙齿特征识别与三维模板可视化设计

## 目标

新版目标从“把普通口腔照片拼成真实二维全景图，再生成三维模型”调整为：

通过上牙弓、下牙弓的多张口腔照片识别牙齿级异常特征，并把这些特征映射到标准三维牙弓模板上，形成可交互查看的用户牙齿情况三维可视化模型。

这个版本优先解决“用户看得懂自己的牙齿问题在哪里”，不承诺还原用户真实牙齿几何尺寸。真实牙冠形态、咬合关系、修复设计仍然属于口扫或专业三维重建范畴。

## 路线判断

当前二维拼接路线的难点在于牙弓是曲面，普通照片来自不同角度，使用单应性或顺序拼接容易把曲率误解释成透视变形，导致完整牙弓图严重畸变。新版不再把完整二维拼接作为核心前置条件，而是把照片当成特征证据来源。

新的主链路是：

1. 采集上牙弓或下牙弓的局部照片。
2. 分割牙齿主体并提取单颗牙实例。
3. 判断牙位或候选牙位。
4. 检测龋齿疑似和缺牙两类核心特征。
5. 多张图片证据融合，生成牙齿级诊断事实。
6. 将事实映射到标准三维牙弓模板。
7. 在三维模型中用隐藏、颜色、贴花、标签或透明度表达异常。

二维拼接保留为辅助预览和证据浏览，不再作为三维展示的强依赖。

## MVP 范围

第一版只做稳定、可展示、可解释的功能：

- 牙弓：上牙弓、下牙弓分开处理。
- 牙位：优先支持 FDI 编号，成人恒牙 11-18、21-28、31-38、41-48。
- 特征：缺牙、龋齿疑似。
- 模型：使用标准牙弓三维模板，每颗牙是独立可控制对象。
- 输出：三维模板状态 JSON、证据截图索引、可视化 GLB/HTML 页面。

暂不做：

- 精确口扫级牙冠几何重建。
- 正畸测量、咬合分析、牙冠修复设计。
- 单张照片直接判断全部牙位。
- 无证据情况下自动生成医学结论。

## 系统结构

### 1. 采集层

复用现有 `pages/capture.py` 的上牙弓、下牙弓流程。拍照指导仍然按左侧、中部、右侧扫描，但这些区域只作为提示和证据覆盖度，不作为硬性拼接门槛。

每张图片需要保留：

- `arch_label`: upper 或 lower。
- `capture_region`: left、center、right 或 unknown。
- `acceptance_score`: 当前已有可接受度评分。
- `image_id`: 后续证据追踪 ID。

### 2. 感知层

复用现有分割能力，新增“牙齿实例与异常特征”输出。

输出分成两类：

- `ToothObservation`: 单张图中看到的某颗牙或候选牙。
- `FeatureObservation`: 单张图中检测到的特征证据。

牙位判断可以分三步推进：

1. 先按图内从左到右排序，得到局部序列。
2. 结合采集区域和上/下牙弓推断候选 FDI 区间。
3. 多图融合后再确定最终牙位。

### 3. 融合层

同一颗牙可能出现在多张图里，融合层负责合并证据。

规则建议：

- 缺牙需要更严格证据，不能只因为某张图没拍到就判断缺失。
- 龋齿疑似特征可以用多图最高置信度加权。
- 同一牙位冲突时保留冲突状态，并在 UI 中显示“需复核”。
- 每个最终特征都必须保留来源图片和置信度。

### 4. 模板层

标准三维牙弓模板需要满足：

- 每颗牙是独立 mesh 或独立节点。
- 每颗牙有固定 `tooth_id`，使用 FDI 编号。
- 每颗牙可独立隐藏、变色、贴局部纹理或显示标签。
- 缺牙通过隐藏牙体或灰色半透明占位表达。
- 龋齿疑似通过表面颜色或标记表达。

模板资产建议目录：

- `assets/dental_templates/adult_upper.glb`
- `assets/dental_templates/adult_lower.glb`
- `assets/dental_templates/template_manifest.json`

### 5. 可视化层

前端展示建议分为三块：

- 左侧：上/下牙弓、牙位列表、特征筛选。
- 中间：三维牙弓模型。
- 右侧：选中牙齿的证据图片、置信度、检测结果。

当前 Streamlit 可以先实现基础版。后续如果要更强交互，可以把三维查看器独立成 Three.js 页面，再由 Streamlit 嵌入。

## 数据契约

核心对象：

- `TemplateTooth`: 模板中一颗牙的定义。
- `FeatureObservation`: 单张图片中的异常证据。
- `ResolvedToothFeature`: 融合后绑定到某颗牙的最终特征。
- `TemplateRenderState`: 三维模板最终渲染状态。

渲染状态 JSON 示例：

```json
{
  "arch_label": "upper",
  "template_id": "adult_upper_v1",
  "teeth": [
    {
      "tooth_id": 16,
      "visible": true,
      "features": [
        {
          "type": "caries_suspected",
          "surface": "occlusal",
          "severity": "medium",
          "confidence": 0.76,
          "evidence_image_ids": ["upper_0007", "upper_0008"]
        }
      ]
    },
    {
      "tooth_id": 14,
      "visible": false,
      "features": [
        {
          "type": "missing",
          "surface": "whole_tooth",
          "severity": "high",
          "confidence": 0.88,
          "evidence_image_ids": ["upper_0003", "upper_0004"]
        }
      ]
    }
  ]
}
```

## 和现有代码的关系

可以复用：

- `pages/capture.py`: 采集入口、上/下牙弓状态、自动拍照质量反馈。
- `dental_stitcher_v1/photo_quality.py`: 图片可接受度。
- `dental_stitcher_v1/segmentation.py`: 牙齿主体分割。
- `dental_stitcher_v1/calibration/instance_extractor.py`: 牙齿实例结构的初始基础。
- `dental_stitcher_v1/calibration/geometry_constraints.py`: 牙弓序列和异常实例过滤思路。

需要新增：

- `dental_stitcher_v1/template3d/schema.py`: 模板和特征数据契约。
- `dental_stitcher_v1/template3d/tooth_map.py`: FDI 牙位和模板节点映射。
- `dental_stitcher_v1/template3d/feature_fusion.py`: 多图证据融合。
- `dental_stitcher_v1/template3d/render_state.py`: 生成三维模板渲染状态。
- `dental_stitcher_v1/template3d/assets.py`: 模板 manifest 加载和校验。

## 分阶段实施

### 第一阶段：模板数据契约

建立 FDI 牙位、特征类型、特征证据和模板渲染状态。先用规则和手工模拟数据跑通从 JSON 到三维模板状态的链路。

验收标准：

- 能生成上牙弓、下牙弓的标准牙位列表。
- 能把缺牙、龋齿疑似等结构化特征映射到牙位。
- 能导出稳定的 `template_render_state.json`。

### 第二阶段：特征检测入口

把现有牙齿分割结果扩展为实例级输出，先支持人工确认或半自动标注，再逐步接模型检测。

验收标准：

- 每张图可以输出候选牙位和异常特征。
- 每条特征有置信度和证据图片。
- 缺牙判断不会因为照片覆盖不足而误判。

### 第三阶段：三维模板查看器

接入标准牙弓 GLB 模板，实现牙齿节点级高亮、隐藏、标签和证据联动。

验收标准：

- 点击牙齿可以查看对应异常和证据图。
- 缺牙和龋齿疑似需要有两种清晰不同的视觉表达。
- 上牙弓和下牙弓可以分别导出结果。

### 第四阶段：替换旧主流程

应用主流程从“分割 -> 拼接 -> 3D 生成”调整为“采集 -> 特征识别 -> 三维模板可视化”。拼接保留为辅助预览。

验收标准：

- 用户不需要等待二维全景拼接成功，也能进入三维展示。
- 结果页以牙位和特征为中心，而不是以拼接图为中心。
- 输出文件包含三维模板状态、证据报告和可视化模型。

## 风险与约束

- 牙位判断是最大风险。普通照片缺少严格位姿，必须允许“候选牙位”和“需复核”状态。
- 缺牙判断不能只依赖负证据，需要牙弓覆盖度和相邻牙关系支撑。
- 龋齿等医学判断需要谨慎措辞，MVP 使用“疑似”表达。
- 标准模板不是用户真实几何，UI 和报告中需要明确这是可视化模型。
- 后续如果要诊疗级精度，需要接入口扫或专业影像数据。
