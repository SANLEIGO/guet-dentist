# 口腔内窥镜图像拼接项目技术栈与算法总结

## 1. 项目定位

本项目是一个面向口腔内窥镜场景的图像处理与辅助重建系统，核心目标是把多张牙弓照片转化为结构更清晰、背景更干净、便于后续分析与三维扩展的拼接结果。当前项目的稳定主线是：

`自动采集引导 -> 牙齿分割 -> 特征匹配 -> 图像配准 -> 牙齿区域融合 -> 结果诊断 -> 3D扩展/模板可视化`

从代码结构上看，当前真正运行的主干在 `dental_stitcher_v1` 目录，前端入口是 `app.py`，拍照采集页是 `pages/capture.py`，三维模板演示页是 `pages/template3d_demo.py`。

## 2. 主要技术栈

### 2.1 开发语言与框架

- Python：项目主开发语言
- Streamlit：构建可交互的 Web 演示界面
- FastAPI / Uvicorn：用于本地 Hunyuan3D 服务化接口

### 2.2 图像处理与数值计算

- OpenCV：图像读写、颜色空间转换、特征提取、配准、GrabCut、单应性估计、图像融合等
- NumPy：矩阵与数组计算
- scikit-image：分片仿射变形 `PiecewiseAffineTransform`
- SciPy：优化求解，服务于自动标定和畸变参数估计

### 2.3 深度学习相关

- PyTorch / TorchVision：U-Net 分割模型推理
- Ultralytics YOLOv8：AlphaDent 牙齿分割模型、龋齿检测模型
- Kornia：作为视觉处理依赖的一部分

### 2.4 三维与可视化

- `model-viewer`：在 Streamlit 页面中预览 GLB 模型
- trimesh：三维模型处理依赖
- Hunyuan3D-2：用于从预处理后的牙弓图像进一步探索 3D 生成

## 3. 系统架构概览

项目整体可以分成四层：

### 3.1 交互层

- `app.py`：主页面，负责上传图片、参数控制、拼接结果展示、Hunyuan3D 服务面板
- `pages/capture.py`：自动拍照采集页面，负责上下牙弓引导、自动保存、照片下载
- `pages/template3d_demo.py`：模板化三维展示页面，负责全景图异常分析与 FDI 牙位映射展示

### 3.2 核心算法层

- `segmentation.py`：牙齿区域分割
- `features.py`：特征提取与描述子匹配
- `registration.py`：图像配准
- `blending.py`：牙齿区域融合
- `pipeline.py`：将各步骤串成完整拼接流水线

### 3.3 质量控制与采集辅助层

- `photo_quality.py`：照片质量打分
- `capture_region.py`：左侧、中部、右侧区域的粗识别
- `auto_capture.py`：自动抓拍逻辑与进度评估
- `diagnostics.py`：输出拼接诊断信息

### 3.4 三维扩展层

- `threed/preprocess.py`：把拼接结果预处理成适合 3D 服务的输入
- `threed/hunyuan_client.py`：通过 HTTP 调用 Hunyuan3D 服务
- `template3d/*`：基于标准牙弓模板做异常可视化与牙位映射

## 4. 核心算法与实现思路

### 4.1 自动采集与质量评估

项目不是简单让用户手动点拍照，而是做了“自动引导 + 自动抓拍”。

### 关键做法

- 采用上下牙弓分阶段采集
- 每个牙弓再按照左后牙、左中段、前牙、右中段、右后牙等顺序给出引导
- 只有当前画面满足质量条件时，系统才自动保存

### 质量评估指标

`photo_quality.py` 中主要使用了以下指标：

- 清晰度：Laplacian 方差
- 亮度与曝光：统计有效内容区域的平均亮度、过曝比例、欠曝比例
- 有效内容占比：排除黑边后估计真正有牙齿信息的区域比例
- 取景质量：估计牙弓是否居中、主体是否过近或过远
- 颜色有效性：过滤全黑、全白、近似灰度的异常图像

最终会融合成一个 `acceptance_score`，用于判断当前照片是否“可接受”。

### 自动抓拍判定

`auto_capture.py` 在质量分数之外，还加入了三个实用约束：

- 稳定性：当前帧与上一帧相比是否足够稳定
- 新颖性：当前帧与上一张已保存图片是否过于相似
- 冷却时间：避免极短时间内连续保存大量重复图

这使得系统更像“半自动采集助手”，而不是单纯的视频截图器。

### 4.2 牙齿区域分割

项目当前支持两种分割方案：

### 方案一：AlphaDent（默认主方案）

- 基于 YOLOv8 的牙齿分割模型
- 输入图像先转为 RGB，再进行推理
- 将实例掩膜合并成整体牙齿区域掩膜

### 方案二：U-Net（可选）

- 使用 PyTorch 推理
- 适合作为另一种语义分割方案
- 当用户显式选择 U-Net 时使用

### 分割后处理

为了提升边界质量，项目加入了：

- CLAHE 对比度增强：提升低对比度图像的牙齿可见性
- GrabCut 精修：对初始掩膜边界进行细化
- 孔洞填充：让牙齿区域更连贯

因此，这一模块是“深度学习粗分割 + 传统图像处理精修”的组合思路。

### 4.3 特征提取与特征匹配

拼接部分并不是直接做端到端生成，而是采用经典视觉拼接路线。

### 特征提取

`features.py` 中支持：

- SIFT
- ORB
- AKAZE

其中当前默认方案是 `SIFT`。

### 为什么默认选 SIFT

- 口腔图像存在视角变化、亮度变化、局部反光
- SIFT 对尺度和旋转更稳定
- 项目中还进一步做了 RootSIFT 处理，提高描述子的鲁棒性

### 具体增强策略

- 先对灰度图做 CLAHE
- 使用牙齿掩膜约束特征提取区域，减少背景干扰
- 对掩膜做膨胀，避免边缘信息丢失
- 使用 Lowe 比率检验筛选匹配点
- 对 SIFT 额外加入正反向一致性检查

这一步的核心思想是：尽量只在“牙齿有效区域”上找稳定特征。

### 4.4 图像配准

匹配点得到后，系统需要估计图像之间的几何变换关系。

### 主方法

- 单应性矩阵 `Homography`
- 使用 RANSAC 抵抗错误匹配

### 回退策略

如果单应性不稳定，系统不会直接失败，而是尝试：

- 全仿射变换
- 部分仿射变换
- 平移模型

### 评价指标

`registration.py` 中使用了：

- 内点数 `inlier_count`
- 内点率 `inlier_ratio`
- 重投影误差 `reprojection_error`

再结合变换合理性判断，决定当前配准结果是否可信。

### 4.5 图像融合

配准完成后，并不是简单覆盖，而是进行牙齿区域的融合。

### 基础融合方式

- 只融合分割出的牙齿区域
- 根据两张图的掩膜距离变换结果计算权重
- 使用高斯平滑形成柔和过渡带

这种方式本质上属于“基于掩膜的羽化融合”。

### 增强融合方式

当内点控制点足够多时，项目还尝试：

- 分片仿射变换 `PiecewiseAffineTransform`

它比单一全局单应性能更好地适应牙弓这种非完全平面、带弧度的目标。

### 4.6 多图顺序拼接与质量门控

对于多张图像，项目采用顺序式拼接策略，而不是一次性全局优化。

### 实现思路

- 先以第 1 张图作为当前参考
- 依次尝试与下一张图拼接
- 每一步都单独做分割、匹配、配准、融合和质量判断
- 质量不达标的图像会被跳过

### 质量门控

`pipeline.py` 中会综合以下信息做 gating：

- 匹配点数量
- 内点数与内点率
- 重投影误差
- 掩膜重叠程度
- 变换后画布面积比例

这样做的优势是可解释性强，失败时可以知道问题出在匹配、配准还是融合。

### 4.7 自动标定与畸变校正

这是项目中一个比较有研究意味的增强模块，目的是缓解口腔内窥镜畸变对拼接质量的影响。

### 基本流程

1. 基于 AlphaDent 提取牙齿实例
2. 将牙齿中心点拟合为牙弓曲线
3. 使用抛物线模型描述牙弓几何约束
4. 通过相邻图像的实例对应关系估计相机外参
5. 估计径向/切向畸变参数
6. 使用类似 Bundle Adjustment 的方式继续优化
7. 对图像进行去畸变，再送入拼接流程

### 用到的算法思想

- 最小二乘抛物线拟合
- 本质矩阵估计相对位姿
- L-BFGS-B 优化畸变参数
- 基于几何一致性的约束优化
- 两轮迭代标定

这一部分体现了项目并不只停留在“二维图像拼接”，而是在尝试把口腔场景的先验几何知识引入系统中。

### 4.8 3D 扩展：Hunyuan3D 服务桥接

项目中还做了从拼接结果走向三维生成的扩展接口。

### 输入预处理

`threed/preprocess.py` 主要做：

- 根据掩膜裁剪有效区域
- 保留最大连通区域
- 适当留白 padding
- 把目标内容放到正方形画布中央
- 生成白底或透明底 PNG

其目的是把拼接结果变成更适合 3D 生成模型处理的输入格式。

### 伪多视图构造

`threed/mv_adapter.py` 还实现了一个比较有工程特色的步骤：

- 由单张牙弓图像合成 `front / left / right / back` 伪多视图
- 通过透视变换和轻量明暗调整，构造出多视角输入
- 生成可下载的多视图 PNG 包和元数据

因此，这里并不是假装已经拥有真实多相机采集，而是用“单张牙弓图 + 几何变换”去近似生成多视图条件输入。

### 服务调用

`threed/hunyuan_client.py` 通过 HTTP 与本地 Hunyuan3D 服务通信，支持：

- 健康检查
- 单图提交
- 多视图提交
- 轮询任务状态

因此这里的 3D 路线更准确地说是：

`拼接图像 -> 掩膜预处理 -> 伪多视图构造 -> Hunyuan3D 服务生成`

### 4.9 三维模板可视化与牙位映射

除了生成式 3D 路线，项目还提供了另一条更稳妥的三维表达方式：把二维检测结果映射到标准牙弓模板上。

### 主要思路

- 对单张全景图做牙齿候选区域提取
- 粗略判断是上牙弓还是下牙弓
- 把横向牙弓区域划分成 16 个槽位
- 依据各槽位占用率估计缺牙候选
- 用龋齿检测模型找到疑似病灶区域
- 将结果映射到 FDI 牙位编号
- 在标准牙弓模板上以高亮、隐藏、标签等方式展示

### 这里用到的算法

- 基于亮度和颜色阈值的牙齿候选区域提取
- 基于位置分布的上/下牙弓启发式判断
- 基于槽位占用率的缺牙候选估计
- 基于 YOLOv8 的龋齿检测

### 工程价值

这种方案虽然不是真正的精确三维重建，但在答辩里很有价值，因为它体现了：

- 从二维检测结果到三维表达的闭环
- 医学牙位编号 FDI 的结构化映射
- 可解释、可视化、便于医生复核的人机交互思路

## 5. 项目中的典型“传统视觉 + 深度学习”融合思想

这个项目的一个鲜明特点，不是完全依赖单一模型，而是把传统图像处理和深度学习组合使用：

- 用深度学习做牙齿区域分割和病灶检测
- 用 OpenCV 做匹配、配准、融合
- 用启发式规则做采集引导、区域判断、牙位槽位估计
- 用几何模型和优化方法做自动标定

这种融合策略的优点是：

- 可解释性更强
- 调试更方便
- 对小样本、真实拍摄波动场景更实用

## 6. 项目亮点

答辩时可以重点强调以下几点：

### 6.1 面向真实采集场景

项目考虑了口腔内窥镜拍摄时常见的问题，如抖动、曝光不稳、黑边、重复拍摄、左右区域切换等，不是只做离线算法。

### 6.2 强调“只对牙齿区域”处理

无论是分割、特征提取还是融合，都尽量围绕牙齿主体进行，减少背景对结果的干扰。

### 6.3 兼顾效果与可解释性

系统输出的不只是最终拼接图，还包括：

- 匹配可视化
- JSON 诊断信息
- 每步质量评估结果

这对工程落地和后续优化非常重要。

### 6.4 具有向 3D 扩展的接口

项目已经不局限于 2D 拼接，而是向两条 3D 路线延伸：

- 生成式 3D：接入 Hunyuan3D
- 模板式 3D：二维异常结果映射到标准牙弓模型

## 7. 当前边界与不足

答辩时建议主动说明边界，这样更显得客观。

### 7.1 主干成熟度最高的是 2D 拼接链路

目前最稳定、最完整的是：

`自动采集 -> 分割 -> 特征匹配 -> 配准 -> 融合 -> 诊断`

3D 相关部分已经打通方向，但仍属于扩展和演示性质。

### 7.2 多图拼接仍以顺序式为主

当前并不是 COLMAP 那样的全局多视图优化框架，而是顺序拼接加质量门控，因此在大视角变化、强弧形牙弓场景下仍可能累计误差。

### 7.3 自动标定模块仍然是简化实现

虽然已经引入相机模型、几何约束和优化过程，但距离严格医学级标定还有进一步完善空间。

### 7.4 模板三维展示更偏“结构化可视化”

模板页的价值主要是异常展示和牙位映射，而不是逐牙精确重建。它更适合辅助表达与结果解释。

### 7.5 LoFTR 仍是预留项

界面和文档里出现过 LoFTR 选项，但当前代码主线实际落地的是 SIFT/ORB/AKAZE 方案，默认和最稳定的是 SIFT。

## 8. 适合答辩时直接说的总结

可以这样概括这个项目：

“这个项目的核心不是单纯做一张全景图，而是围绕口腔内窥镜场景，构建了一条从自动采集引导、牙齿区域分割、特征匹配拼接，到质量诊断和三维扩展的完整流程。在算法上，我采用了深度学习和传统视觉相结合的方法。前者主要负责牙齿分割和龋齿检测，后者负责特征匹配、RANSAC 配准、掩膜融合和质量控制。除此之外，项目还探索了基于牙弓几何约束的自动标定，以及将二维异常映射到标准三维牙弓模板上的可视化方案。” 

## 9. 如果老师追问“你的创新点是什么”

可以从这四点回答：

1. 不是只做离线拼接，而是把自动采集引导和照片质量评估前移到了数据入口。
2. 不是对整张图做粗暴拼接，而是先做牙齿分割，再进行面向牙齿主体的特征匹配和融合。
3. 在传统拼接流程上，加入了牙弓几何约束和自动标定思路。
4. 把二维检测结果进一步延伸到了三维生成和三维模板可视化两个方向。

## 10. 如果老师追问“为什么不用纯深度学习端到端完成拼接”

可以这样回答：

- 口腔图像数据通常较少，端到端模型训练成本高，泛化不稳定。
- 传统特征匹配和几何配准在小样本场景下更可控、更容易调试。
- 该项目更强调工程可解释性和迭代效率，因此采用“深度学习负责感知，传统视觉负责几何处理”的分工更合适。
